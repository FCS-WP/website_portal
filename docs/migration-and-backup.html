1. Concepts & what's at risk

Three persistent stores back the portal. Treat each as a separate failure domain.

Store Lives at Survives make down? Survives disk failure?
Postgres database (epos_portal, ~11 MB) Docker named volume website_portal_postgres-data ✅ Yes ❌ No — losing the laptop = losing the DB unless backed up
Plugin zip storage (storage/app/private/) Bind-mounted to ./portal/storage/ on host ✅ Yes ❌ No — same disk
Encryption key (VAULT_MASTER_KEY) ./portal/.env (gitignored) ✅ Yes ❌ No — and NOT in any backup
Lose VAULT_MASTER_KEY and the DB is half-bricked. Every encrypted column (sites.api_key_encrypted, site_smtp_settings.password_encrypted, vault credentials) becomes permanently unrecoverable — even with a perfect DB restore. Store this value in a password manager the moment you set it, separately from the repo.

2. Database migrations

Laravel 12 migrations live in portal/database/migrations/. Naming is the standard YYYY_MM_DD_HHMMSS_description.php pattern. Every migration runs inside epos-app; you don't invoke php artisan on the host.

2a. Day-to-day migration cycle

Create a new migration:

$ make artisan cmd="make:migration add_x_to_sites_table"

Edit the file under portal/database/migrations/, then apply:

$ make migrate

That runs php artisan migrate inside epos-app. Output prints each batch; success looks like:

   INFO  Running migrations.

  2026_06_04_120000_add_x_to_sites_table .................. DONE

Always check the status afterwards if you migrated anything non-trivial:

$ make artisan cmd="migrate:status"

2b. Full reset (migrate:fresh)

Drops every table, re-runs all migrations from scratch, and re-seeds.

$ make fresh                  # migrate:fresh --seed + clear caches
$ make migrate-fresh           # same, no caches cleared
Destructive. make fresh wipes everything: sites, users, plugin versions, deployment history, activity logs. Take a backup first if anything matters: 
$ make backup        # dumps to portal/backups/<today>/
$ make fresh

2c. Rolling back a single migration

If a migration was bad but you haven't committed yet:

$ make artisan cmd="migrate:rollback --step=1"

Edit the migration file, then make migrate again. If the rollback also broke (e.g. you renamed a column and the down() doesn't reverse cleanly), restore from the most recent backup instead:

$ make backup-list
$ make restore DATE=2026-06-04

2d. Migrating production safely

  1. Always back up before touching prod. Production migrations are not auto-rollback-safe. 
    $ make backup
  2. Migrate in a maintenance window if the migration is long-running. Put the app in maintenance mode first: 
    $ make artisan cmd="down --secret=please-let-me-in"
$ make migrate
$ make artisan cmd="up"
  3. Use --force in production CI (Laravel refuses to migrate in production without it): 
    $ make artisan cmd="migrate --force"
  4. Test the rollback first on a copy of prod data restored to dev. down() methods are easy to write wrong.

3. Backups

3a. What gets backed up

Every backup produces a directory under portal/backups/<YYYY-MM-DD>/ containing three files:

File What Size (typ.)
epos_portal.sql.gz pg_dump --format=plain of the entire DB (schema + data), gzipped. ~17 KB now, will grow with sites/orders
storage.tar.gz Tarball of storage/app/private/ — plugin zip files uploaded to the repository. ~140 KB now
MANIFEST.txt Small text file: timestamp, sizes, restore command. Human-readable proof that a backup ran. < 1 KB
Not in the backup: portal/.env (and therefore VAULT_MASTER_KEY), the agent plugin source, Redis cache, queued jobs in-flight. The .env file is excluded deliberately — secrets in a backup file are a leak vector. Manage it separately.

3b. Manual backup

One-shot backup of the current state:

$ make backup

Output:

Backing up to: /var/www/portal/backups/2026-06-04
  ✓ DB dump: epos_portal.sql.gz (17.5 KB)
  ✓ storage: storage.tar.gz (136.7 KB)
Backup complete.

Options:

$ make artisan cmd="db:backup --skip-storage"       # DB only (faster)
$ make artisan cmd="db:backup --retention=30"       # keep 30 days instead of 14

3c. Scheduled (daily) backup

The scheduler entry in portal/routes/console.php fires db:backup daily at 03:30:

Schedule::command('db:backup')->dailyAt('03:30')->withoutOverlapping();

This runs inside the epos-scheduler container, which must be up:

$ make workers-up       # queue + scheduler containers
$ make workers-logs     # tail to confirm cron fires

To test the scheduled task without waiting for 03:30:

$ make schedule-run     # runs whatever is due right now

Retention. Default is 14 days. The artisan command itself prunes directories named YYYY-MM-DD older than that. Anything else in portal/backups/ is left untouched.

3d. Listing & locating backups

$ make backup-list

Backups live on disk at:

./portal/backups/<YYYY-MM-DD>/
  epos_portal.sql.gz
  storage.tar.gz
  MANIFEST.txt

Inspect a backup without restoring:

$ cat portal/backups/2026-06-04/MANIFEST.txt
$ gunzip -c portal/backups/2026-06-04/epos_portal.sql.gz | head -50
$ tar -tzf portal/backups/2026-06-04/storage.tar.gz | head

4. Restore from a backup

The make restore target handles the full sequence: terminate active DB sessions → drop & recreate the database → pipe the SQL dump back in → un-tar storage → clear Laravel caches. It prompts you to type the date as confirmation so you don't restore by accident.

$ make restore DATE=2026-06-04

About to RESTORE portal/backups/2026-06-04 — this will overwrite the current database.
Type the date to confirm: 2026-06-04
Terminating active sessions (Laravel keeps connections open)...
Dropping + recreating database...
Restoring DB dump...
Restoring private storage...
   INFO  Application cache cleared successfully.
Restore complete. Verify with: make tinker

After restore, sanity-check a few row counts:

$ make tinker
>>> App\Models\Site::count();
>>> App\Models\Plugin::count();
>>> App\Models\DeploymentJob::count();
If anything is connected to the DB at restore time (e.g. queue workers, a Tinker shell, a stray psql), the restore terminates those sessions with pg_terminate_backend. The connections die mid-statement; in-flight queries are rolled back. Idempotent jobs will retry; non-idempotent ones (e.g. a half-sent Telegram message) may be lost. Restore during a quiet window if you can.

5. Disaster recovery scenarios

Scenario A — Accidentally ran make fresh, DB is empty

$ make backup-list                  # find most recent backup
$ make restore DATE=2026-06-04

You're back to that morning's 03:30 state. Anything created after that is lost.

Scenario B — A migration corrupted production data

  1. Take a fresh backup of the current (broken) state for forensics: make backup.
  2. Restore the previous day's backup: make restore DATE=...
  3. Fix the migration locally, test against a copy.
  4. Re-apply: make migrate.

Scenario C — Docker volume removed (docker volume rm)

Bring the stack back up:

$ make up
$ make migrate                       # initial schema
$ make restore DATE=2026-06-04    # overlays the dump

Scenario D — Laptop dead / disk failure

The Docker volume and portal/backups/ are both gone. Recovery requires off-site copies of:

  1. The full repository (from GitHub: git clone).
  2. The portal/backups/ directory — only available if you set up Tier 2 sync.
  3. VAULT_MASTER_KEY — from your password manager. Without this, every encrypted credential is permanently unrecoverable, even after a perfect DB restore.

If you don't have Tier 2 yet, see the next section.

6. Recommended: off-site sync (Tier 2)

Daily local backups protect against accidental data loss; they do not protect against disk failure or stolen hardware. For real disaster recovery, mirror portal/backups/ to an off-host destination.

Cheapest reliable option: Backblaze B2 + rclone. Costs about $0.006/GB/month — for the current backup size (~150 KB/day, 14 days = ~2 MB), this is effectively free.

Setup (one-time):

$ sudo apt install rclone
$ rclone config                       # create remote "b2-backup"

Daily sync (add to host cron, OR to the Laravel scheduler with a shell-out):

# /etc/cron.d/epos-backup-sync — runs after the 03:30 backup completes
35 3 * * * shin cd /home/shin/Documents/My\ data/Develop/Services/website_portal && rclone sync portal/backups/ b2-backup:epos-portal-backups/ >> /var/log/epos-backup-sync.log 2>&1

Restore from off-site:

$ rclone copy b2-backup:epos-portal-backups/2026-06-04/ portal/backups/2026-06-04/
$ make restore DATE=2026-06-04
Equally important: also store VAULT_MASTER_KEY off-host. A password manager entry next to your other infrastructure secrets is fine. This single value is the difference between "lost everything for two hours" and "lost everything forever."

7. Troubleshooting

pg_dump: error: connection to server ... failed

The epos-postgres container isn't running, or DB_HOST isn't postgres. Check:

$ docker ps | grep postgres
$ grep DB_HOST portal/.env       # expect: postgres (not localhost)

dropdb: database "epos_portal" is being accessed by other users

The restore Make target already terminates other sessions, but if a stale Tinker shell or queue worker reconnects between termination and drop, you can hit this. Stop workers first:

$ make workers-down
$ make restore DATE=...
$ make workers-up

ERROR: unrecognized configuration parameter "transaction_timeout" during restore

Harmless. pg_dump 17 (from the apt package in epos-app) emits one SET transaction_timeout line that Postgres 15 doesn't recognize. The line is skipped, the restore continues. Don't chase it.

Backup runs but no files appear on host

Older symptom: backups landed at /var/www/backups inside the container (not bind-mounted). Fixed in the current code — backups go to portal/backups/ via base_path('backups'), which is the bind-mounted ./portal/ mount. If you see this regress, check the path in app/Console/Commands/DatabaseBackup.php.

Scheduler isn't running the daily backup

$ make ps                          # is epos-scheduler running?
$ make workers-logs                # tail for "db:backup" entries
$ make schedule-run                # force-run due tasks now

If epos-scheduler isn't up, no scheduled tasks fire — including sites:ping, vulnerability scans, etc. On prod, make up-prod stacks the worker profile automatically; on dev you need make workers-up explicitly.

8. Things NOT to do

  • Don't rm -rf portal/backups/ in any cleanup script. The directory is gitignored, easy to wipe by accident. If you must clean up, use --retention=N on the command.
  • Don't store portal/.env in a backup. It contains VAULT_MASTER_KEY in plaintext — a leaked backup file would expose every encrypted credential. The backup deliberately skips it. Keep it in a password manager.
  • Don't run make fresh without backing up first. It's idempotent for code (re-runs migrations) but destructive for data. Take the 5 seconds: make backup && make fresh.
  • Don't migrate:fresh on prod. Ever. Use migrate only.
  • Don't restore in the middle of a deployment — let any in-flight WpOrgPluginJob or PushPluginToSite finish first, otherwise sites end up in inconsistent state versus the restored portal view.
  • Don't run pg_dump against prod from your laptop bypassing the artisan command. The Make target handles password env-var safety, locking, and storage layout. Ad-hoc dumps end up in places you can't find later.
  • Don't add migrations that can't be rolled back (e.g. DROP COLUMN with no down() body). If the change truly is one-way, write that explicitly in the down() method as a thrown exception so the rollback fails loudly.
