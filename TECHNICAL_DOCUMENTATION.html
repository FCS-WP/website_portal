§ 01

Project Overview

EPOS Portal is a centralized WordPress management platform that consolidates site administration, plugin lifecycle, credential vaulting, deployment orchestration, and security monitoring into a single operations cockpit.

Tech stack at a glance

Backend
Laravel 12

PHP 8.2-FPM · Sanctum auth · Spatie permissions · Horizon-ready queues.

Frontend
Next.js 16.2.6

React 19.2.4 · TypeScript · Tailwind v4 · shadcn/ui · Zustand store.

Data layer
PostgreSQL 15 · Redis 7

Relational primary store with Redis for cache, queues, and ephemeral state.

Agent
EPOS WP Agent 1.1.2

WordPress plugin exposing REST hooks for sync, deployment, and security telemetry.

Runtime
Docker Compose

Eight services orchestrated through dev/prod/worker profiles via Makefile targets.

Edge
Nginx · Cloudflare Tunnel

Reverse-proxied PHP-FPM with edge tunnels for production exposure.

Three-tier architecture

Tier 1 · Presentation
Next.js Portal
React 19 · TypeScript · Tailwind v4 · shadcn/ui · Zustand · App Router
▼   HTTPS · Bearer Token (Sanctum)   ▼
Tier 2 · Application
Laravel API
PHP 8.2 · 24 controllers · 15 services · 6 jobs · Sanctum + Spatie roles
▼   HTTPS · X-Agent-Key Header   ▼
Tier 3 · Edge node
EPOS WP Agent
WordPress plugin · 18 PHP classes · Custom REST namespace
Design intent
Each tier owns its data and never bypasses the boundary. The Portal UI never contacts WordPress directly — every site action flows through the Laravel API, which holds credentials, audit, and policy.
§ 02

Architecture Overview

Communication patterns

All inter-tier traffic is REST-over-HTTPS. The Portal authenticates as a user; the API authenticates the Agent through a shared header secret; the Agent never originates state-changing calls without a Portal handshake.

EdgeDirectionAuthNotes
Browser → Portal↔SSR / RSCNext.js server components · public assets
Portal → Laravel→Bearer · SanctumPer-user PAT issued at login
Laravel → Agent→X-Agent-KeyPer-site secret stored in sites
Agent → Laravel→X-Agent-KeyHandshake · ping · security reports
Worker → Agent→X-Agent-KeyAsync deployment jobs from queue

Authentication topology

// Portal users — interactive sessions
POST /api/login           → Sanctum issues PAT
Authorization: Bearer <token>
                          → auth:sanctum middleware
                          → EnsureUserIsActive
                          → RoleMiddleware (admin|dev|mkt)

// WordPress agents — service-to-service
X-Agent-Key: <site.agent_key>
                          → AgentAuthMiddleware
                          → resolves $request->site
                          → bypasses session/csrf

State management · Zustand

// store/auth.ts
export const useAuth = create<AuthState>()(
  persist(
    (set) => ({
      user: null,
      token: null,
      hydrated: false,
      setSession: (token, user) => set({ token, user }),
      clear: () => set({ token: null, user: null }),
    }),
    { name: 'epos.auth', storage: createJSONStorage(() => localStorage) }
  )
);

Authentication flow

1. User submits /login form
2. auth.service.login() POSTs /api/login        → Sanctum PAT
3. useAuth.setSession(token, user)               → persisted to localStorage
4. AppLayout reads useAuth on mount, awaits hydration flag
5. If !user → router.replace('/login')
6. Else → fetcher attaches Authorization: Bearer <token> on every call
7. 401 from API → clear() + redirect to /login

Key rotation
Each successful handshake refreshes last_seen_at. Rotating agent_key from the Portal silently invalidates the agent’s next outbound call — the new key must be deployed via wp-cli option update or pushed through the agent settings page.

§ 10

Security Architecture

Security is layered: separate credentials per actor, encrypted secrets at rest, role-gated routes, and continuous telemetry from each managed site.

Authentication methods

Portal users
Sanctum PAT

Bearer tokens issued at /api/login, persisted by Zustand to localStorage, attached to every Portal→API call. EnsureUserIsActive rejects suspended accounts.

WordPress agents
X-Agent-Key header

Per-site secret stored on sites.agent_key, exchanged in AgentAuthMiddleware. Rotatable; survives WP credential changes.

Daily upstream feed sync → SiteVulnerability matches by plugin / WP version. Resolutions flow back through VulnerabilityController@resolve.

FileIntegrityService

Agent class-security-file-monitor hashes WP core / plugins / themes; Portal stores baselines and surfaces drift as FileIntegrityFinding.

Agent batches successful and failed login events; Portal classifies via KnownLoginIp trust level and surfaces anomaly alerts.

2FA detection

Reports presence of common 2FA plugins and enforcement status. Drives the SecurityScore 2FA dimension.

Threat-model note
Loss of any single agent key compromises only that site. Loss of APP_KEY compromises every encrypted column — rotate via artisan key:rotate with a planned re-encryption migration.
